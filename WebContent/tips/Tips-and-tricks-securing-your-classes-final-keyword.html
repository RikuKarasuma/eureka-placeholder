Tips and Tricks: Securing your classes. The Final keyword in Java.


One of the most overlooked principals in development among students and junior developers is scope. Namely class and package scope. In this article we will be going over how the Final keyword should be used to prevent unwanted access to classes or some methods within those classes.

I know what some of you are thinking. Wait can't we just access all classes, methods and variables using reflection anyway so wouldn't this be moot. Well yes, by default if you didn't create a security manager to protect against unauthorised reflection use then your code is at huge risk even with access protections such as the keyword final. Always remember to create a security manager at the beginning of your application.

Now because the final keyword can be used four different ways, I'm going to state below those ways before we get into the meat of things.

  • A final class can't be sub-classed.
  • A final method can't be overridden.
  • A final variable can only be instantiated once.
  • When an inner class is defined within a method, you must make all local method variables it uses from outside that inner class as final.

We will be focusing on just the first two. Which if not used could lead to malicious implementations of that class if malicious code ever got within that JVM.

There are two reasons to get familiar with the word Final. One is to prevent access to the classes and method/s and two is to signal to other developers coming after you that a class, method or variable is immutable by design and should not be tampered with unless necessary.
